# Work OS 使用说明

## 1. 这个项目是什么

这是一个基于 Markdown 文件的个人工作管理系统。

目标是让 AI 或 Agent 根据长期背景、当前计划、历史日志和任务状态，帮助我完成以下工作：

- 生成日计划
- 生成周计划
- 生成月度或阶段计划
- 根据自然语言反馈生成日报或周报
- 维护任务状态，帮助判断下一步安排

这个系统当前处于手动运行阶段，重点是先把闭环跑通，而不是一开始就做复杂自动化。

闭环如下：

- 周计划 -> 日计划 -> 用户反馈 -> 日报 -> 任务状态更新 -> 次日计划调整

## 4. 目录说明

- `00_inbox/`
  - 临时记录、草稿、待整理输入
- `01_context/`
  - 长期稳定背景信息
- `02_state/`
  - 当前任务状态和需要持续维护的事实信息
- `03_plans/`
  - 日计划、周计划、首月或月度计划
- `04_logs/`
  - 日报、周报、月报或阶段复盘

## 5. 每次开始时优先阅读哪些文件

如果你是一个新的模型，且没有聊天上下文，请优先按这个顺序阅读：

1. `README.md`
2. `02_state/current_context.md`
3. `01_context/role.md`
4. `01_context/projects/project_main.md`
5. `02_state/task_index.md`
6. 根据 `02_state/current_context.md` 中记录的当前月计划、当前周计划和最近日志继续阅读
7. 如果需要生成当天计划，再看对应日期的日计划

## 6. 你应该怎么帮助我

你主要承担以下职责：

- 早上：
  - 根据角色背景、项目背景、任务总表、当前周计划、最近日报，生成当天日计划
- 晚上：
  - 根据我给出的自然语言反馈，生成当日日报
- 在必要时：
  - 更新 `02_state/task_index.md`
  - 调整当前周计划中的状态或本周重点
- 每周结束时：
  - 生成周复盘
  - 判断本周计划完成情况
  - 生成下周计划

## 7. 输出要求

- 全部使用中文
- 不要编造未发生的工作进展
- 日计划尽量控制在 3 到 5 个可执行小项
- 计划应尽量小、具体、可打勾
- 日报应如实反映实际进展，即使当天产出不多也可以正常记录
- 如果信息不足，应先基于现有文件做合理整理，再尽量少问问题

## 8. 更新规则

- 新的日计划写入 `03_plans/daily/`
- 新的周计划写入 `03_plans/weekly/`
- 新的首月或月度计划写入 `03_plans/monthly/`
- 新的日报写入 `04_logs/daily/`
- 新的周报写入 `04_logs/weekly/`
- 新的月报或阶段复盘写入 `04_logs/monthly/`
- 当任务真实状态变化时，要同步更新 `02_state/task_index.md`
- 每次执行指令后，要同步更新 `02_state/current_context.md`

## 9. 当前命名约定

- 周计划使用日期区间命名，例如：
  - `2026-04-21_to_2026-04-24.md`
- 首月或阶段计划也使用日期区间命名，例如：
  - `2026-04-21_to_2026-05-15.md`
- 不优先使用自然月第几周或 ISO 周编号，避免和真实入职节奏不一致
- 计划的日期范围优先按工作日划分，而不是按自然周或自然月平均切分
- 如果遇到节假日、调休或明显不上班的日期，应跳过这些日期，再按实际工作日生成周计划和日计划
- 因此某一周可能只有 4 天，某一阶段的第一周或最后一周也可能不是完整 5 天

## 11. 模型工作原则

- 优先依据已有 Markdown 文件做判断
- 优先保持结构一致，不随意改目录口径
- 发现计划与真实进展不一致时，应优先更新事实记录
- 如果发现当前文件结构还不够用，可以提出改进建议，但不要一次性大改
- 当前系统最重要的是跑通闭环，而不是追求形式完整
